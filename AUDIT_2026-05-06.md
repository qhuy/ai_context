# Audit ai_context — 2026-05-06

> Scope : audit logique du template `ai_context`, avec vérification des docs officielles Claude Code et Codex au 2026-05-06.
> Aucun correctif applicatif appliqué dans ce rapport.

## Verdict

Le projet n'a pas besoin d'une refonte totale. Le socle est cohérent : source unique `.ai/`, contexte lean, feature mesh, index JSON, hooks Claude, scripts de contrôle, dogfooding et smoke-test sérieux.

En revanche, il faut une refonte ciblée de la couche **Codex**. Le template documente encore Codex comme un agent essentiellement passif (`AGENTS.md` + scripts manuels + git hooks), alors que Codex dispose maintenant officiellement de hooks, skills, config `.codex`, workflows et mémoire locale. Cette évolution rend l'architecture actuelle sous-optimale pour l'efficacité de l'application context.

## Ce qui marche bien

- `.ai/index.md` impose un Pack A lean et évite de charger quality gate, worklogs, skills et catalogues au démarrage.
- `doctor`, `check-shims`, `check-features`, `measure-context-size` passent localement.
- Le feature mesh est maintenant durci : regex `id/scope`, validation `touches`, `depends_on`, cycles, `touches_shared`, freshness staged.
- Les scripts `ai-context.sh mission/status/brief/review/ship-report` donnent déjà une bonne surface agent-agnostique.
- Les skills Codex `.agents/skills/*` existent et pointent vers les workflows canoniques `.ai/workflows/*`.

## Findings

### P0 — Codex officiel n'est pas exploité

`README.md:64-68` marque encore Codex comme sans `UserPromptSubmit`, sans `PreToolUse`, sans `PostToolUse`, sans `Stop`, sans skills `/aic-*`. C'était vrai historiquement, mais les docs Codex actuelles exposent hooks et skills. Le template ne génère aucun `template/.codex/config.toml.jinja` ni `template/.codex/hooks.json.jinja`.

Impact : Codex doit utiliser `brief <path>` manuellement, alors qu'il pourrait recevoir la même injection juste-à-temps que Claude via hooks repo-local.

Direction : ajouter un agent runtime `codex-runtime` générant `.codex/config.toml` ou `.codex/hooks.json`, conditionné à `codex in agents`.

### P0 — Le hook `features-for-path` n'est pas compatible avec Codex `apply_patch`

`template/.ai/scripts/features-for-path.sh.jinja:166-171` accepte seulement `Write|Edit|MultiEdit` en mode hook. La doc Codex indique que les éditions passent par `apply_patch` et que les matchers peuvent utiliser `apply_patch`, `Edit` ou `Write`, mais l'input garde `tool_name: "apply_patch"`.

Impact : même si on ajoutait un hook Codex `PreToolUse`, il sortirait immédiatement sans injecter de feature context.

Direction : accepter `apply_patch` et extraire les paths du patch. Pour le premier incrément, au minimum intercepter `tool_input.command` et parser les lignes `*** Update File:`, `*** Add File:`, `*** Delete File:` du format `apply_patch`.

### P1 — Drift documentaire sur `AGENTS.md.jinja`

`CHANGELOG.md:24` annonce un `AGENTS.md.jinja` enrichi avec "Setup Commands", "Testing Instructions", "Code Style", "PR Instructions", "Resume cross-session". Or `template/AGENTS.md.jinja:1-15` reste un shim minimal.

Impact : Codex n'a pas les commandes critiques dans son instruction racine, alors que le changelog promet cette efficacité.

Direction : soit enrichir réellement `AGENTS.md.jinja`, soit corriger le changelog. La meilleure option est d'enrichir le shim sans casser le Pack A : sections courtes, commandes exactes, et renvoi vers `.ai/index.md`.

### P1 — README contradictoire sur les ports natifs

`README.md:73` dit que Cursor MDC scopé est livré, puis `README.md:75` dit que "Cursor MDC, AGENTS.md enrichi" sont encore en réflexion. C'est un drift de roadmap.

Impact : le statut produit est difficile à lire ; on ne sait pas si la parité Cursor/Codex est réalisée, en cours ou abandonnée.

Direction : remplacer par un tableau "livré / partiel / proposé" par agent.

### P1 — La mesure de contexte ne couvre pas le coût réel JSON/hook

`measure-context-size.sh` mesure le format texte. Le hook réel Claude appelle `pre-turn-reminder.sh --format=json` dans `template/.claude/settings.json.jinja:8-10`.

Impact : l'estimation est utile mais sous-estime le wrapper et ne prépare pas au format Codex qui aura aussi une enveloppe hook.

Direction : ajouter un mode `--format=json` ou un breakdown "text payload / JSON hook payload / approx tokens".

### P2 — Les wrappers skills Codex sont nombreux et peu hiérarchisés

`.agents/skills/` expose à la fois des skills intentionnels (`aic-frame`, `aic-status`, `aic-review`, `aic-ship`) et des primitives (`aic-feature-new`, `aic-feature-update`, etc.). Codex skills utilisent un budget initial limité et raccourcissent/omettent des descriptions quand il y en a beaucoup.

Impact : risque de matching implicite moins fiable, surtout si l'utilisateur installe d'autres skills.

Direction : regrouper les primitives sous un seul skill `aic-feature` ou les réserver à l'invocation explicite, et renforcer les descriptions des 4-5 skills intentionnels.

## Refonte proposée

### R0 — Réalignement immédiat

- Corriger `README.md` capacité Codex : hooks et skills sont possibles officiellement, mais non encore scaffoldés par ce template.
- Corriger le drift `AGENTS.md.jinja enrichi`.
- Ajouter une feature doc `workflow/codex-runtime-hooks` avant implémentation.

### R1 — Codex Runtime natif

- Ajouter `template/.codex/config.toml.jinja` avec `[features] codex_hooks = true`.
- Ajouter hooks Codex :
  - `SessionStart` ou `UserPromptSubmit` → `bash .ai/scripts/pre-turn-reminder.sh` en texte/additionalContext Codex.
  - `PreToolUse` `^apply_patch$|Edit|Write` → `features-for-path.sh` version multi-path.
  - `PostToolUse` `^apply_patch$` → `auto-worklog-log.sh` compatible patch.
  - `Stop` → `auto-worklog-flush.sh` + `auto-progress.sh`.
- Adapter les scripts pour lire les champs Codex (`tool_name`, `tool_input.command`, `cwd`, `model`).
- Ajouter smoke-test de rendu `.codex/*` et tests unitaires de payload hook Codex.

### R2 — Command surface agent-friendly

- Promouvoir `ai-context.sh` comme CLI agent-friendly :
  - options `--json` sur `status`, `brief`, `review`, `ship-report`.
  - sorties stables avec codes d'erreur documentés.
  - mode dry-run/draft-before-write pour les actions de réparation.
- Enrichir `AGENTS.md.jinja` avec commandes exactes, sans charger de docs lourdes.

### R3 — Observabilité et adoption

- `doctor.sh` doit vérifier `.codex/config.toml` / `.codex/hooks.json` si Codex est sélectionné.
- `check-shims.sh` doit vérifier que `AGENTS.md` mentionne `.ai/index.md`, les commandes clés, et n'inclut pas de charge on-demand.
- `measure-context-size.sh` doit afficher payload texte, payload hook Claude, payload hook Codex.

## Priorité recommandée

1. P0 : rendre `features-for-path.sh` compatible `apply_patch`.
2. P0 : scaffolder `.codex/config.toml` minimal avec `PreToolUse apply_patch`.
3. P1 : enrichir réellement `AGENTS.md.jinja`.
4. P1 : mettre à jour README/PROJECT_STATE/CHANGELOG pour arrêter le drift.
5. P2 : rationaliser les skills Codex.

## Conclusion

La bonne refonte n'est pas de changer le feature mesh. C'est de déplacer Codex du mode "agent non-hooké qui appelle des scripts à la main" vers un mode "runtime natif, hooké, mesurable", en réutilisant les mêmes scripts que Claude. Cela préserve l'architecture existante et augmente fortement l'efficacité sans multiplier les sources de vérité.
